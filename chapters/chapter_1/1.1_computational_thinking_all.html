1.1 Computational Thinking (Combined)

This page combines sections 1.1.1 and 1.1.2. Use the sidebar or sections below to navigate the content.

Using computational thinkingA thinking process involving breaking down problems, finding patterns, focusing on important details, and creating step-by-step solutions. helps us understand complex problems and figure out how to create solutions. These solutions need to be clear enough for a computer, a person, or both to follow.

When solving digital problems, three main strategies are often used:

  • Top-down
  • Bottom-up
  • Modularisation

Three Problem Solving Strategies

Top-Down Approach

This strategy starts with the main problem and breaks it down step-by-step (decompositionBreaking a complex problem into smaller, easier-to-handle parts.) into smaller sub-problems or modulesSelf-contained parts of a program or system. until each part is simple enough to solve easily.

  • Breaks big problems into smaller pieces.
  • Uses a step-by-step refinement process.
  • Work flows from the general goal downwards.
  • Often shown as a hierarchy diagram.
  • Common in procedural programmingProgramming style focusing on sequences of instructions (procedures/functions), like in the C language..

Bottom-Up Approach

This approach begins by developing the most fundamental modulesSelf-contained parts of a program or system, often representing basic functionalities. first. These are built and tested independently, then progressively combined (integratedThe process of combining smaller, tested modules or components into larger subsystems or the final complete system.) to form larger subsystems, eventually building up to the complete solution.

  • Starts with basic building blocks/modules.
  • Focuses on combining finished parts (integration).
  • Builds up complexity towards the final goal.
  • Often used with Object-Oriented ProgrammingProgramming style using 'objects' that bundle data and actions (e.g., Python, Java). (OOP).

Modularisation

Modularisation divides a system into separate, independent modulesSelf-contained, often reusable units of code or functionality with a specific purpose.. Each handles a specific task with a well-defined interfaceHow a module interacts with other parts of the system (e.g., inputs it accepts, outputs it produces)., aiming for manageable, reusable, and maintainable components.

  • Divides system into independent modules.
  • Promotes code reusability.
  • Simplifies testing and maintenance.
  • Requires clear interfaces between modules.
  • Modules can often be developed in parallel.

Comparing Top-Down and Bottom-Up

Aspect Top-Down Bottom-Up

Top-Down: Starts big, gets detailed. Good when you know the overall goal clearly. Breaks the problem down (decomposition).

Bottom-Up: Starts small, builds up. Useful when the basic parts are clear, but the final structure might change. Puts pieces together (integration).

Modularisation: Creates reusable, independent blocks. Works well with both, especially OOP. Makes systems easier to manage and update.

What is Decomposition?

DecompositionBreaking down a complex problem or system into smaller, more manageable parts. is a core technique used in computational thinking. It's about taking a large, complex problem and breaking it down into smaller, simpler sub-problems.

Computational thinking relies on four key techniques, often called pillars:

  • Decomposition: Breaking down.
  • Pattern Recognition: Finding similarities.
  • Abstraction: Focusing on essentials.
  • Algorithms: Step-by-step solutions.

By breaking a problem down, each smaller part becomes easier to understand, manage, and eventually solve.

Real-World Insight

Think about building a large application like a social media platform. Developers don't try to build the whole thing at once! They decompose it into modules like user authentication, news feed generation, messaging, profile management, etc. Each module can then be developed and tested more easily.

Decomposition and the Top-Down Approach

As discussed in the Problem Solving Approaches section above, the Top-DownA problem-solving approach starting with the overall problem and breaking it down into smaller parts. strategy heavily relies on decomposition.

You start with the main problem and keep breaking it down into smaller tasks or modulesSelf-contained units or components of a larger system, often resulting from decomposition. until they are simple enough to tackle independently. Once these individual modules are created (e.g., coded), they are combined (integratedThe process of combining smaller, tested modules or components into larger subsystems or the final complete system.) to form the solution to the original, larger problem.

Pros and Cons of Decomposition

Advantages

  • Teamwork: Different people can work on separate modules simultaneously.
  • Easier Maintenance: Updates/fixes can target specific modules.
  • Manageability: Makes complex problems less daunting.
  • Reusability: Well-defined modules can be used elsewhere.
  • Testing: Easier to test smaller, individual modules.

Disadvantages

  • Integration Challenges: Modules might not fit together correctly.
  • Misunderstanding: Incorrect decomposition if the initial problem isn't fully grasped.
  • Design Overhead: Defining clear interfaces takes effort.
  • Potential Redundancy: Similar logic might appear in different modules.

Four Steps of Decomposition

When breaking down a problem for coding, these steps help guide the process:

Clearly define what the problem is and what processes are involved. Use language specific to the problem's context (e.g., retail terms for a shop system, physics terms for a simulation).

Example: Problem is "Calculate total cost for items in a shopping basket". Processes involve getting item prices, quantities, applying discounts, and summing up.

Use decomposition (often top-down) to split the main problem into smaller, distinct tasks and subtasks. Keep breaking down until each task is simple and focused. Ensure all tasks directly contribute to solving the original problem.

Example: Break down "Calculate total cost" into: 1. Get item details (price, quantity). 2. Calculate subtotal per item. 3. Apply discounts. 4. Calculate final total.

Document each task clearly. What does it do? What information does it need (inputs)? What result does it produce (outputs)? This documentation helps developers implement the solution correctly.

Example: Document "Apply discounts": Needs item price and discount rules as input. Outputs the discounted price.

Share the decomposed structure and task descriptions with the team. Assign responsibility for different modules. Clear communication ensures everyone understands their part and how it fits into the whole solution before combining the modules.

Example: Team member A takes the "Get item details" module, Team member B takes "Apply discounts", etc. Regular check-ins ensure compatibility.

Decomposition in Code (Python Example)

Decomposition often translates into creating separate functions or methods in code. Consider calculating the area and perimeter of a rectangle:

# --- Without Decomposition ---
def process_rectangle(length, width):
    area = length * width
    perimeter = 2 * (length + width)
    print(f"Area: {area}")
    print(f"Perimeter: {perimeter}")

process_rectangle(10, 5)

# --- With Decomposition ---
def calculate_area(length, width):
    return length * width

def calculate_perimeter(length, width):
    return 2 * (length + width)

def display_results(area, perimeter):
    print(f"Calculated Area: {area}")
    print(f"Calculated Perimeter: {perimeter}")

# Main part uses the decomposed functions
rect_length = 10
rect_width = 5
rect_area = calculate_area(rect_length, rect_width)
rect_perimeter = calculate_perimeter(rect_length, rect_width)
display_results(rect_area, rect_perimeter)

The decomposed version separates concerns: one function calculates area, another calculates perimeter, and a third displays results. This makes the code easier to read, test, and reuse.

Connecting the Pillars

Decomposition is often the first step in computational thinking. Once a problem is broken down:

  • You can look for patternsIdentifying similarities or repeated processes within or between the decomposed parts. across the smaller sub-problems.
  • You can use abstractionFocusing on the essential features of the sub-problems while ignoring irrelevant details. to focus on what's important in each part.
  • You can then design specific algorithmsCreating step-by-step instructions to solve each of the smaller, well-defined sub-problems. to solve each sub-problem.

These techniques work together to tackle complexity effectively.
